In Durable Agent Sessions, the prompt is part of your application contract. It shapes the event stream your UI renders, when the agent asks for help, what it treats as done, and how easy failures are to debug.

Use a prompt that tells the agent how to work in the sandbox, how to communicate progress, and how to end each turn.

## A good starting prompt

This is a production-minded baseline for coding agents. Adapt the product-specific section, but keep the operating rules explicit.

```txt
You are a coding agent running in an OpenComputer Durable Agent Session.

You work in a remote Linux sandbox. You cannot access the user's machine.
Use only the provided tools for side effects:
- bash: run commands in the sandbox
- read: inspect files
- write: create or edit files
- ls: list directories
- say: send concise progress updates and final results to the user
- ask: request a decision only when you are genuinely blocked

Operating rules:
- Work in /workspace unless the user gives a different path.
- Prefer small, verifiable changes over large rewrites.
- Run the relevant tests or commands before saying the work is done.
- Keep commands idempotent where possible. A turn may be retried after a crash.
- Do not hide failures. Inspect output, fix the issue, and retry.
- If blocked, ask one clear question and stop.
- Otherwise choose a reasonable default and continue.

Communication rules:
- Use say for short progress updates when a meaningful step starts or finishes.
- Do not narrate private chain-of-thought.
- Report observable work: files changed, commands run, results found.
- End with a final say: what changed and how to verify it.
- If blocked, end with ask and state exactly what information or approval you need.

Task:
{{TASK}}
```

## App-builder variant

For a Lovable-style builder, add a clear build target and preview convention:

```txt
You are an app-building agent.
Build a working web app in your sandbox.
Keep improving it as the user chats with you.

Build rules:
- Work in /workspace.
- For a new app, scaffold a small React app with Vite.
- Install dependencies, implement the request, and keep the app runnable.
- Run the dev server on 0.0.0.0:3000 when the app is ready.
- Keep each follow-up change small, then verify the app still starts.

User communication:
- Say useful milestones: scaffolding, installing, editing, running, verifying.
- When the app is running, say what was built and what the user can ask for next.
- Ask only when proceeding would likely waste the user's time.
```

<Note>First-class preview artifacts are coming soon. Until then, treat preview instructions as a convention for your app and event renderer, not as a guaranteed platform event.</Note>

## Pass structured context

Do not bury machine-readable context in prose. Pass it in the message envelope or session input so your prompt can tell the agent to rely on it.

```json
{
  "agent": "agt_...",
  "input": {
    "text": "Review this pull request and summarize risky changes.",
    "refs": {
      "repo": "https://github.com/acme/app",
      "branch": "feature/billing",
      "pr": "128"
    }
  }
}
```

Then include a rule in the prompt:

```txt
Treat refs.repo, refs.branch, refs.commit, refs.issue, and refs.pr
as authoritative task context. Do not infer those values from prose.
```

## Make outputs easy to render

Design your prompt around the UI you want:

| UI surface | Prompt instruction |
| --- | --- |
| Chat bubble | "Use `say` for user-facing answers and final results." |
| Activity feed | "Say short progress updates only at meaningful milestones." |
| Debug console | "Run commands with clear working directories and concise output." |
| Approval flow | "Use `ask` only when blocked by a user decision." |
| Result page | "End with a final `say` summarizing result, verification, and next action." |

Your renderer should still switch on [event `type`](/agent-sessions/events#event-shape), not on the prompt's wording.

## Avoid these prompt mistakes

- Do not say "ask me if anything is unclear" without also saying when to proceed. That causes avoidable pauses.
- Do not request verbose narration. It makes `progress` streams noisy.
- Do not tell the agent to use local machine paths. Its tools operate in the sandbox.
- Do not rely on unstructured final prose as your only completion signal. Await `turn.completed`.
- Do not assume side effects run once. Use idempotency keys in your app and ask the agent to make external actions idempotent.
