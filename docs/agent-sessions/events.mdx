Every step a session takes is an **event**, appended to the log in order. The log is the source of truth ‚Äî you read it, stream it, and resume from any point. Everything you build on top ‚Äî a chat UI, a live "what's it doing" feed, notifications, an audit trail ‚Äî comes from reading or streaming these events and filtering them by **level**.

```json
{ "id": "evt_‚Ä¶", "seq": 12, "ts": "‚Ä¶", "session": "sess_‚Ä¶",
  "actor": { "id": "agent", "type": "agent" },
  "kind": "message", "level": "user",
  "body": { "text": "3 tests fail in auth.spec.ts" }, "refs": {} }
```

- **`id`** ‚Äî the event's own unique id (`evt_‚Ä¶`). The client idempotency key for steering is separate (deduped per session ‚Äî see [steering](/agent-sessions/messaging)).
- **`kind`** ‚Äî `message` (content), `status` (lifecycle), or `error`.
- **`actor`** ‚Äî who produced it: `{ id, display?, type: "human" | "agent" | "system" }`.
- **`level`** ‚Äî visibility; see below.
- **`body` / `refs`** ‚Äî payload and related ids.

Inbound and outbound messages share one [envelope](/agent-sessions/messaging#the-envelope) shape.

## Levels ‚Äî pick what to show

Every event carries a `level` saying **who it's for**. You filter the stream (and webhook deliveries) by it, so each surface shows the right amount of detail.

| Level | What lands here | Reach for it when you want‚Ä¶ |
| --- | --- | --- |
| `user` | What a person is meant to read ‚Äî the answer, results, a question back to them. The default for webhook delivery. | a chat bubble, the final result, a notification |
| `progress` | Plain-language step updates: "cloning the repo", "running tests", "found 3 failures". | a live "what's it doing right now" feed or status line |
| `internal` | Tool calls and runtime detail ‚Äî verbose, not meant for end users (the model's private reasoning is never surfaced). | debugging, an audit trail, your own internal console |

Filtering is **cumulative** ‚Äî `user` is the narrowest, `internal` is everything:

- `?level=user` ‚Üí just `user`
- `?level=progress` ‚Üí `progress` + `user`
- `?level=internal` ‚Üí everything

So a customer-facing chat asks for `user`; an agent-activity view asks for `progress`; debugging your agent asks for `internal`.

## seq, cursor, head

- **`seq`** ‚Äî a monotonic id assigned to each event; the cursor space.
- **`head`** ‚Äî the highest `seq` in the session.
- **`cursor`** ‚Äî how far a reader (or the runtime) has consumed.

Read or stream with `after=<seq>` and you get everything since that point. This is how **resume** works: reconnect with the last `seq` you saw ‚Äî you miss nothing and get no duplicates. In the browser, native `EventSource` does this automatically ‚Äî each event's `seq` is the SSE `id`, so it resumes via `Last-Event-ID` on reconnect (see the [quickstart](/agent-sessions/quickstart)).
