A **Repo** connects a Git repository to your sessions and draws a **security boundary**
around it. The agent reads and edits the checked-out code in its own sandbox, but every
authenticated GitHub operation — clone, fetch, push, PR, comment — runs in a
**separate, single-use sandbox the agent can't reach**, with a token minted just in time
and destroyed after. Registering a repo is how you set that boundary up.

<Note>
**Why isolate.** A GitHub token is *authority*, not just a secret — anything holding it
can push, read your other repos, or exfiltrate. So OpenComputer keeps it out of the
agent entirely: the agent requests GitHub *outcomes*, and the credential lives only in
the repo's dedicated operation sandbox, one operation at a time.
</Note>

A Repo is to *where* an agent works what an [Agent](/agent-sessions/agents) is to *what*
it runs: a reusable, owner-scoped entity you register once and reference by id.

<Note>**Status — in active rollout.** Today a session accepts inline `sources` at
create (below). Registered **Repos**, the broker drop-in, automatic checkout, and the
GitHub tools are landing now; this page is the target contract.</Note>

## What you provide

You own a **GitHub App** (you hold its private key) and host a small **token broker**
endpoint. OpenComputer stores no GitHub credential; at materialization it calls your
broker with a **signed** request, and your broker mints a fresh, repo-scoped,
short-lived installation token for that one operation. Your broker:

1. **Verifies the request is OpenComputer's** — each request is signed; verify it
   against OpenComputer's published JWKS (no shared secret for you to store).
2. **Binds it to your org** — the request carries the OpenComputer `owner_id`, the
   `repo`, and the `operation`. Confirm the `owner_id` is *your* OpenComputer org (an
   allowlist, or a one-time registration handshake) — so another customer can't point
   at your broker URL and have it mint tokens for your installation (confused-deputy).
3. **Mints to the policy, never more** — issue an installation token scoped to that
   repo and the minimum permissions the operation needs, with a short TTL.

<Note>A drop-in `ocGithubBroker()` that does all three is **coming soon**. Until then,
implement the contract above — it's a thin wrapper over your existing GitHub App
client. See the [broker contract](/agent-sessions/api-reference#repos) for exact shapes.</Note>

## Register a repo

<CodeGroup>

```ts TypeScript SDK
const repo = await oc.repos.create({
  owner: "acme",
  repo: "web",
  auth: { type: "token_broker", brokerUrl: "https://app.example.com/oc/github-token" },
});
```

```http REST API
POST /v3/repos
Authorization: Bearer $OPENCOMPUTER_API_KEY

{ "provider": "github", "owner": "acme", "repo": "web",
  "auth": { "type": "token_broker", "broker_url": "https://app.example.com/oc/github-token" } }
```

</CodeGroup>

Get-or-create, owner-scoped, idempotent by `(provider, owner, repo)`. It stores only the
broker URL, repo identity, and policy defaults — **never a token**. Sessions then
reference the repo with no per-session secret.

## Use it in a session

```ts
const session = await oc.sessions.create({
  agent: agent.id,
  input: "Review this pull request.",
  sources: [{ repo: repo.id, ref: "refs/pull/42/head", sha: "abc123…", name: "head" }],
});
// checked out at /workspace/sources/head before the first turn
```

- **`ref`** (required) — the fetch ref (a branch or `refs/pull/N/head`); a private SHA
  can't be fetched directly.
- **`sha`** (required) — the exact commit, pinned and verified after fetch.

A session can list several sources (a PR's base + head). Sources are **immutable**; the
agent pulls more refs mid-session via the `source.fetch_ref` tool.

**Inline, without registering** — pass `url` + a per-session `grant` (a short-lived
reference your app mints) instead of a registered `repo`:

```ts
sources: [{ url: "https://github.com/acme/web.git", ref: "refs/pull/42/head", sha: "abc123…",
  auth: { type: "token_broker", brokerUrl: "…", grant: "<grant>" } }]
```

## What the agent can do

Local Git works normally on the checkout (`status`/`diff`/`log`, edit, test, local
commits). Anything needing the network is a **typed tool** run in the isolated sandbox —
the agent never runs authenticated `git`/`gh` and never sees a token:

| Tool | Does | Status |
| --- | --- | --- |
| `source.fetch_ref` | fetch another ref into the checkout | Coming soon |
| `github.publish_branch` / `publish_pull_request` | push changes / open a PR | Coming soon |
| `github.comment` / `github.set_status` | comment, set a commit status | Coming soon |

Write actions need write scope; if the repo's grant doesn't cover it the action reports
`auth_required` rather than reusing read scope. *(Coming soon.)*

## Guarantees

- **No GitHub token is ever stored** — a repo holds only a broker URL + identity; a
  per-session grant is an encrypted *reference*, not a credential.
- **Tokens are minted just in time, per operation** — repo-scoped, short-lived, and
  destroyed with the isolated job. None ever reaches the agent's prompt, shell, files,
  env, the event log, or telemetry.
- Agent pushes go to an `oc/<session>/…` branch namespace — **no protected-branch push,
  no force-push.**
- Submodules and Git LFS aren't supported yet. Lost-sandbox recovery and prebuilt
  checkouts for large repos ([Workspaces](/agent-sessions/workspaces)) are firming up.

<Note>
**No broker yet?** Pass a real short-lived token inline with the deliberately-named
`risky_short_lived_token` auth. OpenComputer rejects a far-future/missing expiry, stores
it **encrypted** only until the first checkout, then purges it — it never becomes a
durable credential, and the source can't be recovered once the token expires. Prefer the
broker.
</Note>
