title: "Repos & GitHub"
description: "Use GitHub Apps to authorize private repo work while keeping credentials out of agent sandboxes"
---

A **Repo** is repository identity plus policy. A **GitHub App** is the authority
OpenComputer uses to obtain short-lived GitHub tokens. Sessions point at repos and
sources; GitHub credentials stay behind that boundary and never enter the agent sandbox.

<Note>**Preview** — APIs may change before general availability.</Note>

## Model

| Concept | Role | Boundary |
| --- | --- | --- |
| GitHub App | Token authority for GitHub operations. | Does not define repo policy or session refs. |
| Installation | GitHub account and repo selection for an App. | `byo_broker` can keep this state inside your backend. |
| Repo | Provider, owner, repo name, optional App pin, and permission ceiling. | Stores no tokens, private keys, or session refs. |
| Source | Per-session checkout: repo/url, `ref`, `sha`, and local name. | Auth comes from the repo's App unless using inline auth. |

Private GitHub access is used to prepare the source checkout before a turn starts. Inside
the sandbox, the agent works on local files; authenticated GitHub operations stay outside
the model-executed environment.

<Note>
The Repo/Source shape is provider-neutral: `provider`, `owner`, `repo`, `ref`, `sha`.
GitHub uses GitHub Apps as its auth surface. Other providers can add their own auth
surface while sessions continue to reference repos and sources the same way.
</Note>

## GitHub App modes

| Mode | Who owns the App key | Who mints tokens | Use when |
| --- | --- | --- | --- |
| `oc_app` | OpenComputer | OpenComputer | You want private repo access with no App key handling. |
| `byo_stored_key` | You; encrypted in OpenComputer | OpenComputer | You want GitHub actions to appear as your own App without hosting a broker. |
| `byo_broker` | You; kept in your backend | Your broker endpoint | Your backend must own GitHub policy, token minting, and App lifecycle. |

In all modes, each operation asks for the minimum GitHub permission it needs. Checkout
needs read-only contents access; write-capable actions ask for write access only for that
operation.

### OpenComputer GitHub App

Install the **OpenComputer** GitHub App and select the repositories agents may work in —
you choose the repos, and can change the selection or uninstall anytime. No keys to
manage, no endpoint to host. OpenComputer mints a fresh, narrowly-scoped token for each
operation, so durable sessions and recovery work with nothing else from you.

<Warning>
The App may request write-capable GitHub permissions on the repos you select. OpenComputer
does **not** hand that broad authority to the agent: each operation gets a fresh token with
the minimum permission it needs, such as read-only for checkout and write only for a
publish/comment action.
</Warning>

Install the OC-owned App from its published GitHub App install URL. OpenComputer does not
expose an SDK method to mint that URL because the App is owned by OpenComputer. Once it's
installed on a repo, reference that repo in a session's `sources` (below) — OpenComputer
resolves the App and mints a token just in time.

You do not need to enumerate installations to use a repo. OpenComputer resolves the App
automatically when a session starts.

### Bring your own GitHub App

Use a bring-your-own App when you want GitHub audit logs, PRs, comments, and statuses to
come from your own App instead of OpenComputer's.

`byo_stored_key` means OpenComputer stores your App private key encrypted and mints
operation-scoped installation tokens on demand. This keeps setup simple, but OpenComputer
is still the token authority.

`byo_broker` means your backend keeps the App key and owns the full GitHub lifecycle. When
OpenComputer needs a GitHub operation, it sends a signed request with the repo, operation,
ref/sha, and requested permissions; your broker returns a short-lived installation token
or denies the request. OpenComputer may not be able to list installations for `byo_broker`
Apps, because your backend owns that state.

### Inline short-lived token

The no-setup path is to pass a real short-lived token inline on the source, using the
deliberately-named `risky_short_lived_token` auth:

<CodeGroup>

```ts TypeScript SDK
sources: [{
  url: "https://github.com/acme/web.git",
  ref: "refs/pull/42/head",
  sha: "abc123…",
  auth: {
    type: "risky_short_lived_token",
    token: "ghs_…",
    expiresAt: "…",
  },
}]
```

```json REST API
"sources": [{
  "url": "https://github.com/acme/web.git",
  "ref": "refs/pull/42/head",
  "sha": "abc123...",
  "auth": {
    "type": "risky_short_lived_token",
    "token": "ghs_...",
    "expires_at": "..."
  }
}]
```

</CodeGroup>

<Warning>
The inline token **can't refresh** — once it expires the session can't fetch, push, or
recover, so use it only for sessions that finish within the token's life. OpenComputer
rejects a far-future or missing expiry and holds it encrypted only until the first
checkout, then purges it. Use this only when a GitHub App is not configured.
</Warning>

## Register a repo

Once the GitHub App is installed, register a repo — optional: any installed repo can be
used in a session directly; a Repo adds policy and a stable handle.

<CodeGroup>

```ts TypeScript SDK
const repo = await oc.repos.create({
  owner: "acme",
  repo: "web",
  // appId: "gha_...", // optional: org default, then OC App
  defaults: { allow: ["contents:read"] },
});
```

```http REST API
POST /v3/repos
Authorization: Bearer $OPENCOMPUTER_API_KEY

{ "provider": "github", "owner": "acme", "repo": "web" }
```

</CodeGroup>

Get-or-create, owner-scoped, idempotent by `(provider, owner, repo)`. **No credential is
passed** — auth resolves through the GitHub App. Optional `appId` pins a repo to a
specific App; otherwise OpenComputer uses your org default App, then the OpenComputer App.
Optional `defaults`:
`{ branchNamespace, allow }` (`allow` is a GitHub permission ceiling; OpenComputer still
requests the minimum token per operation).

## Use it in a session

<CodeGroup>

```ts TypeScript SDK
const session = await oc.sessions.create({
  agent: agent.id,
  input: "Review this pull request.",
  sources: [{
    repo: repo.id,
    ref: "refs/pull/42/head",
    sha: "abc123…",
    name: "head",
  }],
});
// checked out at /workspace/sources/head before the first turn
```

```http REST API
POST /v3/sessions
Authorization: Bearer $OPENCOMPUTER_API_KEY

{
  "agent": "agt_...",
  "input": "Review this pull request.",
  "sources": [{
    "repo": "repo_...",
    "ref": "refs/pull/42/head",
    "sha": "abc123...",
    "name": "head"
  }]
}
```

</CodeGroup>

- **`ref`** (required) — the fetch ref (a branch or `refs/pull/N/head`); a private SHA
  can't be fetched directly.
- **`sha`** (required) — the exact commit, pinned and verified after fetch.

You can also reference `"owner/repo"` directly instead of a registered id. A session can
list several immutable sources, such as a PR's base and head.

## Check source status

The create response includes a sanitized source snapshot. Poll `session.listSources()` for
live materialization status:

```ts
const sources = await session.listSources();

for (const source of sources) {
  if (source.status === "auth_required" || source.status === "failed") {
    showSourceError(source.errorCode, source.errorMessage);
  }
}
```

Common `errorCode` values: `source.auth_required`, `source.auth_ambiguous`,
`source.repo_not_selected`, `source.permission_missing`, `source.sha_mismatch`, and
`source.timeout`.

## What the agent can do

Local Git works normally on the checkout (`status`/`diff`/`log`, edit, test, local
commits). Anything needing GitHub credentials runs outside the agent sandbox, so the agent
never runs authenticated `git`/`gh` and never sees a token.

## Guarantees

- **GitHub App-backed tokens are never stored** — OpenComputer obtains short-lived,
  repo-scoped tokens just in time and destroys them. None reaches the agent's prompt,
  shell, files, env, the event log, or telemetry. (The one exception is inline-token auth,
  held encrypted only until the first checkout, then purged.)
- App private keys for the OpenComputer App and stored-key Apps live only in
  OpenComputer's **control plane**, never in a sandbox. `byo_broker` App keys stay in your
  backend.
- Platform-mediated pushes use an `oc/<session>/…` branch namespace — **no
  protected-branch push, no force-push.**
- Submodules and Git LFS are not part of the checkout contract.
