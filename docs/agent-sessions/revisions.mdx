You **deploy** an agent's behavior ‚Äî its `prompt`, `model`, and [skills](#skills). Each deploy produces an immutable **revision**; the agent's **active revision** is what new [sessions](/agent-sessions/sessions) use. Roll back by re-activating an earlier revision.

Deploy three ways ‚Äî all the same underneath: the **API/SDK**, the **`oc` CLI**, or [**push to GitHub**](/agent-sessions/deploy-from-github).

<Note>Revisions are **linear**: each explicit deploy appends a new revision (the number only goes up), like a deployment on Vercel/Fly. Rollback moves the **active pointer** back ‚Äî it never rewrites history. In-flight sessions keep the revision they started on; only new sessions pick up a change.</Note>

## Deploy

`POST /agents/:id/deploys` is the one command. `input.type: "inline"` sends behavior directly (API / CLI / dashboard); `input.type: "github"` deploys a linked repo ‚Äî see [Deploy from GitHub](/agent-sessions/deploy-from-github).

An inline payload is the **complete** behavior ‚Äî omitted fields aren't inherited (`prompt` is required; omitted `skills` means *no skills*).

<CodeGroup>

```ts TypeScript SDK
const deploy = await oc.agents.deploys.create("agt_123", {
  input: {
    type: "inline",
    prompt: "You triage pull requests.",
    model: "anthropic/claude-opus-4-8",
    skills: [{ path: "triage/SKILL.md", content: "---\nname: triage\ndescription: Triage a PR\n---\nSummarize the diff‚Ä¶" }],
  },
  activationPolicy: "auto",
});
```

```http REST API
POST https://api.opencomputer.dev/v3/agents/agt_123/deploys
Authorization: Bearer $OPENCOMPUTER_API_KEY
Content-Type: application/json

{
  "input": {
    "type": "inline",
    "prompt": "You triage pull requests.",
    "model": "anthropic/claude-opus-4-8",
    "skills": [{ "path": "triage/SKILL.md", "content": "---\nname: triage\ndescription: Triage a PR\n---\nSummarize the diff‚Ä¶" }]
  },
  "activation_policy": "auto"
}
```

```bash CLI
# from a directory holding agent.toml + prompt.md + skills/
oc agent deploy
```

</CodeGroup>

The response is a **deploy** object:

```json
{ "deploy": { "id": "dep_‚Ä¶", "state": "ready", "revision_id": "rev_‚Ä¶", "activation_policy": "auto" } }
```

Inline deploys finish synchronously (`state: "ready"` with a `revision_id`). Asynchronous deploys (GitHub) return `state: "queued"` ‚Äî poll `GET /agents/:id/deploys/:deploy_id` or watch [events](/agent-sessions/events).

### Activation

`activation_policy` decides what happens once a deploy is ready:

| Value | Effect |
| --- | --- |
| `auto` *(default)* | the new revision becomes **active** immediately |
| `manual` | the revision is **staged** ‚Äî created but not active; [promote](#roll-back-promote) it later |

Staging lets you test before going live: deploy with `manual`, start a session against the staged revision, then promote it. (`auto_after_checks` is reserved for an upcoming eval gate.)

<Tip>Editing `prompt`/`model` via [`PATCH /agents/:id`](/agent-sessions/agents) also deploys a revision ‚Äî a shorthand for small edits.</Tip>

## Roll back / promote

Rollback and promote are the **same primitive** ‚Äî set the active revision. Re-activate any earlier revision (rollback) or a staged one (promote):

<CodeGroup>

```ts TypeScript SDK
await oc.agents.revisions.activate("agt_123", 3);
```

```http REST API
POST https://api.opencomputer.dev/v3/agents/agt_123/revisions/3/activate
Authorization: Bearer $OPENCOMPUTER_API_KEY
```

```bash CLI
oc agent rollback 3
```

</CodeGroup>

Instant ‚Äî no rebuild. Running sessions are unaffected; new sessions use the newly-active revision.

## What's on a revision

| Field | |
|---|---|
| `number` | Monotonic per agent (`1, 2, 3, ‚Ä¶`). |
| `prompt` / `model` | The behavior at deploy time. |
| `skill_bundle_digest` | The skill files (a content-addressed bundle); `null` when there are no skills. |
| `digest` | A content hash of the behavior (used for change detection). |

The agent itself holds only **identity + bindings** (`name`, `runtime`, `credential`, `limits`) plus a pointer to the active revision; `prompt`/`model` are served from that revision.

## Inspect

```http REST API
GET /v3/agents/agt_123/deploys             # deploy timeline (state, source, actor)
GET /v3/agents/agt_123/deploys/dep_123     # one deploy (poll async deploys to ready|failed)
GET /v3/agents/agt_123/revisions           # revision history (newest first; active flagged)
GET /v3/agents/agt_123/revisions/5         # one revision + its skill file manifest
GET /v3/agents/agt_123/activations         # active-pointer audit log
```

`GET /agents/:id` also includes an `active_revision` summary (`id`, `number`, `digest`).

## Skills

Skills ‚Äî reusable instruction folders the agent loads when relevant ‚Äî are **part of a revision**, so they're deployed, versioned, and rolled back exactly like the prompt and model. Include them inline in a deploy (`skills:[‚Ä¶]`, above), upload a `.zip` (`PUT ‚Ä¶/skills` / `oc agent skills set`), or ship a `skills/` directory [from a repo](/agent-sessions/deploy-from-github).

See the **[Skills](/agent-sessions/skills)** page for the `SKILL.md` format, how skills are invoked, all three ways to add them, and limits.

## Not yet supported

These are **coming soon** ‚Äî a deploy that includes them is rejected today:
- **MCP servers** (`mcp.json`)
- **Custom runtimes** (a `runtime/` directory)
- **Skills on `codex` agents** ‚Äî skills are `claude`-runtime only for now

## Dashboard

The agent page in the [dashboard](https://app.opencomputer.dev) mirrors all of this: a **Revisions** panel (history, the active revision, one-click rollback) and a **Skills** panel (current skills, plus upload-`.zip` / remove).
