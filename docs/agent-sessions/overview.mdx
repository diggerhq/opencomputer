title: "Durable Agent Sessions"
description: "Resumable, steerable agent runs — start one in a single API call"
---

Give an agent a prompt and a task; OpenComputer runs it on a managed runtime in a sandbox and records **every step as a durable, streamable, resumable session**. Watch it work live, drop your connection and reconnect right where you left off, let it idle and the sandbox hibernates (≈ free), then steer it with a message or get notified by webhook.

```bash
curl https://api.opencomputer.dev/v3/sessions \
  -H "Authorization: Bearer $OPENCOMPUTER_API_KEY" \
  -d '{ "model": "anthropic/claude-opus-4-8",
        "prompt": "You are a coding assistant.",
        "input": "Clone github.com/acme/widget, run its tests, and summarize what fails." }'
# → { "session": { "id": "sess_…", "status": "running" }, "client_token": "…" }
```

## Start a run

`POST /v3/sessions` with a `prompt`, `model`, and `input` (or a saved [agent](/agent-sessions/agents)) and the run begins immediately. One call. See the [quickstart](/agent-sessions/quickstart).

## Stream & resume

Tail the [event log](/agent-sessions/events) over SSE from any cursor. Disconnect and reconnect with the last `seq` you saw — you continue exactly where you left off, nothing missed or duplicated.

## Steer & get notified

Send a follow-up [message](/agent-sessions/messaging) any time; the session wakes and continues with full prior context. Register a **webhook** and the agent's user-facing output is pushed to you.

## Browser-safe access

Every session hands back a scoped [client token](/agent-sessions/authentication#client-tokens) (`read` + `steer`) so a front-end can stream and steer **without** your API key.

## What the platform handles for you

The non-trivial parts you'd otherwise build — and get subtly wrong — handled:

<CardGroup cols={2}>
  <Card title="Reliable delivery" icon="paper-plane">
    Outbound messages are delivered at-least-once, idempotent per destination, retried with backoff, and dead-lettered — no lost or duplicated webhooks.
  </Card>
  <Card title="Managed sandbox lifecycle" icon="box">
    Provision, hibernate on idle (storage-only ≈ free), wake in milliseconds, recover a dead sandbox, tear down — none of it your problem.
  </Card>
  <Card title="Isolated model access" icon="shield-halved">
    The model loop runs in a sandbox separate from your agent's code execution, so a prompt-injected agent can't reach your model key.
  </Card>
  <Card title="Durable event log" icon="database">
    Append + notify in one transaction, idempotent and monotonic — the log is the source of truth, so resume is exact and there's no lost-update window.
  </Card>
  <Card title="One driver, crash-safe" icon="lock">
    A fenced lease guarantees a single active runtime; a crashed or stale run is fenced out and re-woken from its cursor.
  </Card>
  <Card title="Budget & safety guards" icon="gauge-high">
    Per-session spend cap, crash-loop backstop, and SSRF-guarded, signed webhook delivery — runaways and abuse are bounded by default.
  </Card>
</CardGroup>

## When to use it

Reach for sessions when you want a long-running or interactive agent and don't want to build the plumbing above yourself. For a one-shot command in a raw box, use a [sandbox](/sandboxes/overview) directly.
