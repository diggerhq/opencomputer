Base URL `https://api.opencomputer.dev/v3`. Management calls authenticate with your **org API key**; stream and steer also accept a **[client token](/agent-sessions/authentication#client-tokens)**.

Each section toggles between the **TypeScript SDK** (`@opencomputer/sdk`), **REST**, and the **`oc` CLI** — pick one and the whole page follows.

```ts
import { OpenComputer, connectSession } from "@opencomputer/sdk";

const oc = new OpenComputer({ apiKey: process.env.OPENCOMPUTER_API_KEY! });
const session = await oc.sessions.get("ses_...");        // server, org key
// or, in a browser:  await connectSession({ sessionId, clientToken })
```

```bash
# the oc CLI authenticates with OPENCOMPUTER_API_KEY (or `oc login`); --json on any command
oc whoami
oc agent deploy        # deploy the agent directory in the cwd
```

<Note>The SDK returns the same objects with **camelCase** fields (`client_token` → `session.clientToken`, `created_at` → `createdAt`, `last_turn` → `lastTurn`, …) and takes camelCase params (`idempotency_key` → `idempotencyKey`). The wire shapes below are the raw JSON. Truly opaque blobs — session `metadata`, event `refs`, event `raw` — pass through **verbatim** both directions; everything else, including event `body`, is camelCased. The CLI prints a human table by default and the raw JSON with `--json`.</Note>

`agent` is **required** (create a saved [agent](#agents) first); the model must resolve to a source — **Managed** (billed to your credits, the default) or a usable [credential](#credentials) — else `422 no_credential` (or `422 managed_unavailable`). A session **pins the agent's active [revision](#deployments-and-revisions)** (prompt/model/skills) for its whole life; pass **`revision`** (a number or `rev_…` id) to pin a specific, non-active one — e.g. to test a [staged](/agent-sessions/revisions#staging-vs-activating) revision. `key` gives get-or-create (one session per key); a request-level `Idempotency-Key` header makes a keyless create retry-safe. `metadata` is opaque **application routing state** — a JSON object (≤ 16 KB) stored on the session, returned by get/list, and delivered **verbatim in webhooks**; never shown to the model, not indexed. `input` is the task — a string or a full [envelope](/agent-sessions/messaging#message-input); **truncate large inputs** and have the agent pull the rest through its [sandbox tools](/agent-sessions/runtime-tools). `limits` `{ tokens, turn_seconds, turns }` are **non-monetary**; hitting one ends the turn with a matching `last_turn.yield_reason`. `sources` checks out repositories **before the first turn** (the checkout credential never enters the sandbox; see [Repos & GitHub](#github-apps-and-repos)) — each entry is `{ repo, ref, sha, name? }` (registered repo / `owner/repo`) or inline `{ url, ref, sha, name?, auth }`; `ref` and `sha` are both **required**.

Client tokens accept `scopes?` (subset of `read`, `steer`) and `ttl?` seconds, clamped 60–86400 (SDK: `ttlSeconds`).

```
Session = { id, status, head, input_cursor,
            agent_snapshot: { prompt_hash, model, runtime, revision },   // revision = the pinned revision number
            agent_id, credential_id,   // resolved + pinned at create
            last_turn: { id, state, yield_reason?, result_event_id? },
            limits: { tokens?, turn_seconds?, turns? },
            metadata?, key?, created_at }

Turn = { id, state, yield_reason?, result_event_id?, error?, started_at?, completed_at? }

SessionSource = { repo, ref, sha, name? } | { url, ref, sha, name?, auth }
SourceAuth    = { type: "risky_short_lived_token", token, expires_at }   // inline only; can't refresh
SourceSummary = { name, status, path, sha, resolved_sha?, error_code?, error_message?, retryable? }
//   status: pending | materializing | resolved | failed | auth_required
//   error_code: source.auth_required | source.repo_not_selected | source.permission_missing | source.mint_failed | source.ref_not_found | source.sha_mismatch
```

`yield_reason` (`completed | needs_input | error | budget_exceeded | deadline_exceeded | max_turns | canceled`) is the machine-readable "why the turn ended" — `needs_input` means the agent asked a question; answer it by [steering](#steer).

Create body: `name`, `prompt`, `model`, `runtime?` (`claude` | `codex`), `key?`, `credential?`, `limits?`. PATCH accepts `prompt?`, `model?`, `key?`, `credential?`, `limits?` — **not `runtime`** (fixed at creation). The `runtime` / `model` provider must agree (`claude`↔`anthropic/…`, `codex`↔`openai/…`), validated at create. `credential` is a closed value: a `cred_…` id (BYO, must match the provider) **or** the literal `"managed"` (run **via OpenComputer, billed to your credits**). `key`/`credential` are mutually exclusive.

`POST /agents` is **idempotent by `name`** — a fresh create returns **201**; a repeat with matching non-secret config returns the existing agent **200**; differing config **409**. Editing `prompt`/`model` (here, or via the [deploy API](#deployments-and-revisions)) creates a new **revision**; the agent carries a pointer to its **active revision** (`GET /agents/:id` includes an `active_revision` summary), and sessions pin it. The agent row itself holds only identity + bindings (`name`, `runtime`, `credential`, `limits`).

## Deployments and revisions

You **deploy** an agent's behavior (`prompt` + `model` + `skills`); each deployment produces an immutable **revision**, and the **active revision** is what new sessions use. See the [Deployments & revisions guide](/agent-sessions/revisions).

`input` is `{ type: "inline", prompt, model?, skills?, runtime? }` (the **complete** behavior — omitted `skills` means none) **or** `{ type: "github", ref?, sha? }` (deploys the agent's [linked repo](#deploy-from-a-repo); `repo`/`path` come from the link). `activate: false` **stages** the revision (test, then promote). Inline deployments are synchronous (`state: "ready"` with a `revision_id`); GitHub deployments are async (`state: "queued"`). `POST /agents/:id/revisions` remains as **inline-only sugar**.

```
Deployment = { id, state, revision_id|null, active, source?, error?, created_at }
//   state: accepted | queued | fetching | validating | uploading | ready | failed | superseded | skipped
//   result: created | skipped | failed
```

## Errors & rate limits

One envelope: `{ "error": { "type", "message" } }`; the SDK throws typed `OpenComputerError` subclasses (`AuthError`, `NotFoundError`, `ConflictError`, `ValidationError`, `RateLimitError`); the CLI exits non-zero and prints the message (the JSON with `--json`). Cross-owner resources return `404` (not `403`). Rate/concurrency limits aren't enforced at launch (a `429` with `Retry-After` may come later; the SDK retries it).
