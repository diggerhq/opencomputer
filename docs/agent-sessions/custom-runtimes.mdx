A custom runtime is a container image for your own agent harness — anything beyond the built-in `claude` and `codex` runtimes. Once registered, select it per agent with `runtime: "<name>"`; sessions, events, steering, and webhooks work the same way.

## Runtime invocation

For each turn, the platform starts your image with the session id, read cursor, events API URL, and a fenced **turn token**. The image must:

1. read the new input from the [event log](/agent-sessions/events) at the cursor;
2. drive its own agent loop;
3. stream events back over the authenticated events API;
4. perform side effects through the remote [sandbox tools](/agent-sessions/runtime-tools);
5. exit 0 when it has nothing left to do.

The turn token is fenced: writes from a superseded runtime instance are rejected. Non-zero exit is treated as a crash and the platform restarts the turn in place.

## Journal and recovery

The image should be stateless. Store resumable state in the provided journal directory. The durable record is the event log plus that journal, so crashes, idle hibernation, and machine restore follow the same [recovery](/agent-sessions/runtimes#failure-behavior) behavior as `claude`.

## Model configuration

The model is configuration, not part of the image. A runtime declares which providers it can drive; switching providers requires a runtime image that supports the target provider.

## Register runtimes

Register a runtime with `oc runtime push <image>` and select it with `runtime: "<name>"`. Runtimes are versioned; each session pins the version it starts with, so pushing a new build never changes in-flight sessions.
