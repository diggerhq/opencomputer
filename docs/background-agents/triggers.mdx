Triggers are how the world reaches your agent; deliveries are how its work
gets back out. Both run through **connections** — integrations OpenComputer
operates for you.

## Connections

```bash
oc connect github            # install the OpenComputer GitHub App on your org
oc connect github --as-user  # optional: personal grant, so deliveries act as you
```

One click installs the GitHub App. You never register an app, configure a
webhook, or handle a secret — events start flowing and the platform can act
on repos you granted.

Model provider keys are connections too — see
[Models](/background-agents/models).

## Triggers

Declared on the agent; each maps an event to a session:

```toml
[[triggers]]
type = "github"
on = "issue.labeled:agent"      # start (or resume) a session per issue
```

Supported GitHub events include issue labels and PR/issue comments. When an
event arrives, the platform acknowledges it instantly, then routes it:

- **Routing keys** group events into sessions — by default, one session per
  issue. Two labels on the same issue reach the same session; different
  issues get parallel sessions.
- **Follow-ups find their thread.** A comment on a PR the agent opened
  resumes the session that opened it — the session remembers everything.

## Deliveries

Agents never hold GitHub tokens. When an agent wants to open a PR or post a
comment, it requests a **delivery**, and the platform performs it:

- With a personal grant (`oc connect github --as-user`), the PR is opened
  **as the person who triggered the work** — so review attribution and
  permissions work exactly as if they'd pushed it themselves.
- Without a grant, the App opens it with clear *"on behalf of @user"*
  attribution.
- Defaults are conservative: draft PRs only, never to protected branches —
  see [policies](/background-agents/agents#policies).

Every delivery appears on the session timeline with what was done and
where, and the resulting PR links back to its session.
