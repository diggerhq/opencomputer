---
title: "Agents"
description: "From a single prompt to a versioned agent you own as code"
---

An agent is configuration, not compute: what it knows (prompt, skills),
what drives it (runtime and model), where it works (workspace), what wakes
it (triggers), and what it's allowed to do (policies). Sessions reference an
agent; the agent itself never "runs."

## Start as a prompt

The minimal agent is one call:

```bash
oc agent create fixer \
  --workspace web \
  --model anthropic/claude-opus-4-8 \
  --prompt "You fix bugs labeled 'agent' in acme/web." \
  --trigger github:issue.labeled:agent
```

```bash
curl -X POST https://api.opencomputer.dev/v2/agents \
  -H "Authorization: Bearer $OC_API_KEY" \
  -d '{
    "name": "fixer",
    "workspace": "web",
    "runtime": { "image": "opencode", "model": "anthropic/claude-opus-4-8" },
    "prompt": "You fix bugs labeled agent in acme/web.",
    "triggers": [{ "type": "github", "on": "issue.labeled:agent" }]
  }'
```

## Grow into a repo

When one prompt isn't enough, the agent becomes a directory you own:

```
my-agent/
  agent.toml        # name, runtime, model, workspace, triggers, policies
  prompt.md
  skills/
    testing.md      # conventions the agent loads when relevant
    deploys.md
  mcp.json          # MCP servers the agent may use
```

```bash
oc agent deploy     # from the directory — like deploying any app
```

Keep it in git, review changes in PRs, deploy from CI — it's code. The API
and the directory are two views of the same thing; nothing requires GitHub.

```toml
# agent.toml
name = "fixer"
workspace = "web"

[runtime]
image = "opencode"
model = "anthropic/claude-opus-4-8"

[[triggers]]
type = "github"
on = "issue.labeled:agent"

[policies]
budget.per_session_usd = 5
```

## Versions

Every change — a config update, a deploy, a model switch — creates a new
immutable version:

```bash
oc agent versions fixer
oc agent diff fixer v3 v4
oc agent rollback fixer v3
```

Sessions pin the version they started with, so updating an agent never
changes the behavior of work already in flight; new sessions pick up the
latest.

## Policies

Policies bound what an agent can do, with safe defaults:

- **Deliveries**: draft PRs only by default; protected branches are never
  pushed to; restrict target repos explicitly.
- **Budget**: per-session and per-month spend caps.
- **Models**: restrict which providers the agent may use.
- **Network**: sandbox egress is allowlist-based.

Organization-level policies act as ceilings — an agent can be stricter than
the org default, never looser. Policy changes are versioned with the agent,
so every loosening is a visible diff.
