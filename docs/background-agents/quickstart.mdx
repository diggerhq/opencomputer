After this page, labeling a GitHub issue `agent` produces a draft PR, and
commenting on that PR wakes the same agent to follow up.

<Note>**Preview** — see the [overview](/background-agents/overview); some steps
below are still rolling out.</Note>

## 1. Connect GitHub

```bash
oc connect github
```

Installs the OpenComputer GitHub App on your org. No app registration,
webhook setup, or secrets. To have PRs open **as you** rather than as the App
on your behalf, also run `oc connect github --as-user`.

## 2. Create a workspace (optional, Preview)

```bash
oc workspace create web \
  --repo github.com/acme/web \
  --setup "pnpm i && pnpm build"
```

A workspace prepares a repo ahead of time: the first build clones it, runs
setup, and snapshots the result, then rebuilds on every push so sessions
start from a ready machine matching the latest code. `oc workspace builds
web` lists the builds. **In v0 this step is not built yet** — skip it and the
session clones the repo on demand.

## 3. Create the agent

```bash
oc agent create fixer \
  --model anthropic/claude-opus-4-8 \
  --prompt "You fix bugs labeled 'agent' in acme/web. Keep diffs minimal. Run the affected tests before pushing." \
  --trigger github:issue.labeled:agent
```

## Use it

Label an issue `agent`. A session starts within seconds:

```bash
oc session list --agent fixer
oc session attach <session-id>   # live events; Ctrl-C detaches, the agent keeps working
```

A draft PR opens, attributed to the person who labeled the issue — as them if
they ran `oc connect github --as-user`, otherwise opened by the App on their
behalf. The session goes dormant; `oc session get <id>` shows `usage.active_seconds` — dormant time
uses no compute.

Comment on the PR: *"also update the changelog."* The session wakes with its
full context, pushes a commit, and sleeps again. The thread persists; a
comment weeks later wakes the same context.

## Change the model

```bash
oc agent update fixer --model openai/gpt-5.x
```

Effective when the agent next wakes — switching providers means the runtime
reconnects; a change within the same provider applies on the very next model
call. Same prompt and triggers. Each change is a new agent
version — `oc agent rollback fixer v1` reverts.

## Or: talk to it in Slack

An agent can be conversational instead of (or as well as) issue-triggered.
Declare a Slack trigger and it becomes a teammate you @mention in a thread:

```toml
# opencomputer.toml
name = "slack-helper"

[runtime]
image = "claude"
model = "anthropic/claude-opus-4-8"

[[triggers]]
type = "slack"
on   = "mention"

[connection]
github = "diggerhq"   # the repos it may act on; it resolves which from the chat
```

```bash
oc connect slack          # install the OpenComputer Slack app (Preview — see below)
oc agent deploy ./slack-helper
```

`/invite` it to a channel and @mention it:

```
you   @slack-helper parseDuration in oc-agent-demo-target mis-parses "1.5h" — fix it?
bot   On it — pulling oc-agent-demo-target…
bot   Found it: the regex silently drops "1.". Throw on bad input, or support fractions?
you   throw
bot   Done — tests pass. 📦 Opened PR #42
```

It works out *which* repo from the conversation, investigates in a sandbox,
**asks** when it needs a decision (and pauses, free, until you reply), and opens
a draft PR — narrating in-thread the whole time. Full walkthrough:
[Channels](/background-agents/channels).

<Note>**Preview** — in v0 the Slack app is platform-configured for a single demo
workspace; self-serve `oc connect slack` is planned.</Note>
