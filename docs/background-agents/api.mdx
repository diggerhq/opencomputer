`Authorization: Bearer $OC_API_KEY`. The CLI is a wrapper over these calls.

## Agents

```http
POST   /v2/agents                    # create — can be as little as {name, connection, runtime, prompt}
GET    /v2/agents                    # list
GET    /v2/agents/:id
PATCH  /v2/agents/:id                # any config change creates a new version
GET    /v2/agents/:id/versions
POST   /v2/agents/:id/versions      # deploy a bundle (what `oc agent deploy` calls)
```

```json
{
  "name": "fixer",
  "connection": "github:acme",
  "runtime": { "image": "claude", "model": "anthropic/claude-opus-4-8" },
  "prompt": "You fix bugs labeled 'agent' in acme/web.",
  "triggers": [
    { "type": "github", "on": "issue.labeled:agent",
      "route": { "key": "issue.{number}", "mode": "reuse" } }
  ],
  "policies": { "budget": { "per_session_usd": 5 } }
}
```

## Workspaces

Keyed by the repo(s) they prepare, not by agent — a session
[resolves](/background-agents/workspaces#resolution-and-sharing) which repo(s)
to work in (from the trigger, the conversation, a default, or by asking) and
matches a workspace covering that repo and ref.

```http
POST   /v2/workspaces                # { name, recipe: {repos: [{repo, ref?}], setup[], warm[]}, refresh, pool }
GET    /v2/workspaces?repo=&ref=     # list / resolve by repo and ref
GET    /v2/workspaces/:id            # recipe, pool status, latest build
GET    /v2/workspaces/:id/builds     # immutable lineage
POST   /v2/workspaces/:id/builds     # build now
```

## Sessions

```http
POST   /v2/sessions                  # { agent, input?, key? } — start explicitly
GET    /v2/sessions?agent=&status=   # list / filter
GET    /v2/sessions/:id              # status, pins, sandbox info, usage
PATCH  /v2/sessions/:id              # { model_override } — this session only, audited
GET    /v2/sessions/:id/events       # ?after=<cursor>&stream=sse&level=user
POST   /v2/sessions/:id/events      # raw append (low-level): [{ kind: "message", body: "..." }]
POST   /v2/sessions/:id/archive
```

`status` is `active | dormant | awaiting_input | archived` (`awaiting_input` =
the agent emitted an `ask` and yielded).

Session object (abridged):

```json
{
  "id": "s_8f2…",
  "status": "dormant",
  "pins": {
    "agent_version": 4,
    "workspace_build": 14,
    "runtime": "claude@sha256:…",
    "model": { "requested": "anthropic/claude-opus-4-8", "served": "anthropic/claude-opus-4-8" }
  },
  "sandbox": { "state": "hibernated", "preview_urls": [] },
  "usage": { "active_seconds": 672, "tokens": { "in": 184211, "out": 9302 }, "usd": 0.47 }
}
```

Event envelope — every event carries a `level`
([visibility class](/background-agents/observability#levels)):

```json
{
  "id": "evt_01…",
  "ts": "2026-06-12T09:31:04Z",
  "actor": "platform",
  "kind": "delivery",
  "level": "user",
  "target": "chan_…",
  "body": { "type": "pr.open", "repo": "acme/web", "number": 413, "as": "maria" },
  "refs": [{ "type": "github_pr", "number": 413 }]
}
```

Kinds: `message`, `say`, `action`, `ask`, `delivery`, `status`, `timer`,
`exec.completed`, `error`. Levels: `internal | progress | user`, derived at
append from `(actor, kind)` with an optional override. `target` (optional) names
a [channel binding](#messaging) to deliver an event to a specific thread.

## Messaging

The two-way surface a chat UI is built on — send messages in, render the stream
out, bind external threads (see [Channels](/background-agents/channels)):

```http
POST   /v2/sessions/:id/messages     # a steer (in): { text, actor?, idem_key? }
GET    /v2/sessions/:id/messages?after=…    # conversation history (replay a UI)
POST   /v2/sessions/:id/channels     # bind a thread: { provider, thread_ref, role[] }
GET    /v2/sessions/:id/channels     # list bindings
```

`POST /messages` is the ergonomic steer (one inbound message); `POST /events`
is the low-level raw append underneath it. Channel `role[]` is
`conversation | delivery`.

<Note>**Preview** — `POST /v2/sessions/:id/interrupt` (mid-turn interrupt
steering) is designed but not built in v0. Steers are consumed at the next turn
boundary; see [Sessions](/background-agents/sessions#preview-mid-turn-interrupt).</Note>

## Connections & models

```http
POST   /v2/connections               # { type: "github" | "github_user" | "slack" | "anthropic" | "openai", ... }
GET    /v2/connections
GET    /v2/models                    # aggregated catalog from connected providers
```

The `slack` connection (per-workspace OAuth, token held platform-side) powers
the [Slack channel](/background-agents/channels). In v0 it is
platform-configured for a single workspace; self-serve `oc connect slack` is
Preview.
