title: "Background Agents"

A background agent is an agent that works without anyone watching: a GitHub
issue gets labeled, the agent picks it up in a fresh machine, fixes the bug,
opens a draft PR, and goes back to sleep. Someone comments on the PR a day
later — the same agent wakes up with full context and pushes a follow-up.
You pay for the minutes it worked, not the days it waited.

Background Agents is the OpenComputer layer that makes this a few commands
instead of an infrastructure project. It handles everything around the agent
loop: receiving events, keeping sessions alive for months, preparing fast
machines, delivering results back to GitHub, and metering cost — while
staying **model-agnostic**: switch the model, or the entire harness, with
one configuration change.

## The pieces

| Concept | What it is |
|---|---|
| [Agent](/background-agents/agents) | Your agent's definition: prompt, skills, model, triggers, policies. Starts as a single prompt; grows into a repo you own. Versioned. |
| [Session](/background-agents/sessions) | A durable thread of work: an event timeline that never loses state. Active while working, dormant (≈ free) while waiting. |
| [Workspace](/background-agents/workspaces) | A prepared machine for your repo — dependencies installed, build warm — kept fresh automatically so sessions start in seconds. |
| [Trigger](/background-agents/triggers) | What starts or resumes sessions: GitHub events routed to the right session automatically. |
| [Connection](/background-agents/triggers#connections) | One-click integrations (GitHub) and model provider keys. No app registration, no webhook config. |
| [Runtime](/background-agents/runtimes) | The loop that drives a session. Pick a default; the model it uses is plain configuration. |

## How it relates to Sandboxes

Background Agents is built on [sandboxes](/sandboxes/overview) — every
session does its work inside a real, isolated Linux VM, booted from your
workspace's latest build. You never manage these sandboxes directly, but
they're ordinary OpenComputer sandboxes underneath: you can inspect one with
`oc session shell` whenever you need to see exactly what the agent sees.

If you want full control of the loop and the machine, the
[Agent SDK](/agents/overview) remains the lower-level path. Background
Agents is for when you want the platform to run the agent for you.

## What makes it different

- **Sessions are durable data.** A session is not a process — compute
  attaches when there's work and detaches when there isn't. Kill anything,
  the session continues from exactly where it was. Threads can stay alive
  for months at ~zero cost.
- **Fast by preparation, not by magic.** Workspaces rebuild themselves on a
  schedule, so a triggered session boots into a machine where dependencies
  are installed and the build is warm — typically ~2 seconds to working.
- **Model-agnostic for real.** Models are addressed by provider-prefixed IDs
  (`anthropic/claude-opus-4-8`, `openai/gpt-5.x`) and changed with one
  command — taking effect on the session's next model call, without
  restarting anything. Bring your own provider keys, including OpenRouter.
- **Results arrive where work happens.** Agents don't hold tokens; the
  platform opens PRs and posts comments on their behalf, attributed to the
  person who asked.
