title: "Background Agents"

A background agent works unattended: an issue gets labeled, the agent fixes
it in a fresh machine and opens a draft PR. A comment on that PR a day later
wakes the same agent with full context. You pay for the minutes it worked.

Background Agents runs everything around the agent loop: event ingestion,
session state, machine preparation, delivery back to GitHub, metering. The
model — or the entire harness — is a configuration change.

## Concepts

| Concept | What it is |
|---|---|
| [Agent](/background-agents/agents) | Definition: prompt, skills, model, triggers, policies. Versioned. Starts as one prompt; can grow into a repo. |
| [Session](/background-agents/sessions) | One thread of work: a durable event timeline. Active while working, dormant otherwise; dormancy is not billed. |
| [Workspace](/background-agents/workspaces) | A machine image for your repo, rebuilt on a schedule — dependencies installed, build warm. |
| [Trigger](/background-agents/triggers) | GitHub events, routed to the right session. |
| [Connection](/background-agents/triggers#connections) | The GitHub App install and model provider keys. |
| [Runtime](/background-agents/runtimes) | The loop that drives a session. Its model is separate configuration. |

## How it works

A session is a durable timeline of events. While there is work, two machines
attach: a **runtime** (the loop) and a **sandbox** (the machine it acts in);
they detach when the work pauses. All state — events, the model
conversation, files — leaves the machines as it is produced, so either can
die at any point and the session resumes where it was. Details:
[How it works](/background-agents/how-it-works).

## Relation to sandboxes

Sessions work inside ordinary [sandboxes](/sandboxes/overview), booted from
the workspace's latest build and managed for you. `oc session shell <id>`
opens the actual machine. For full control of the loop and the machine, use
the [Agent SDK](/agents/overview) instead.

## Properties

- Sessions start in ~2 seconds, because the workspace rebuilt itself
  recently.
- Models are addressed `provider/model` and switched with one command,
  effective on the next call. Bring your own keys, including OpenRouter.
- Agents hold no tokens. The platform opens PRs and posts comments on their
  behalf, attributed to the requesting user.
