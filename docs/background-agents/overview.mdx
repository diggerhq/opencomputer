A background agent works unattended: an issue gets labeled, the agent fixes
it in a fresh machine and opens a draft PR. A comment on that PR a day later
wakes the same agent with full context. You pay for the minutes it worked.

Background Agents runs everything around the agent loop: event ingestion,
session state, machine preparation, delivery back to GitHub, metering. The
model — or the entire harness — is a configuration change.

## Concepts

| Concept | What it is |
|---|---|
| [Agent](/background-agents/agents) | Definition: prompt, skills, model, triggers, policies, and a connection. Versioned. Not bound to a repo or workspace; via its connection it acts across many repos. |
| [Session](/background-agents/sessions) | One thread of work: a durable event timeline. Active while working, dormant otherwise; dormant holds no compute. |
| [Workspace](/background-agents/workspaces) | A prepared environment for one or more repos, rebuilt on every push — dependencies installed, build warm. Not tied to an agent. |
| [Trigger](/background-agents/triggers) | GitHub events or a Slack @mention, routed to the right session. |
| [Channel](/background-agents/channels) | A two-way thread the session talks in — Slack today, a GitHub PR. Bindings are additive; a session is never typed by surface. |
| [Connection](/background-agents/triggers#connections) | The GitHub App install, Slack workspace, and model provider keys. |
| [Runtime](/background-agents/runtimes) | The loop that drives a session. Its model is separate configuration. |

## How it works

A session is a durable timeline of events. While there is work, two machines
attach: a **runtime** (the loop) and a **sandbox** (the machine it acts in);
they detach when the work pauses. All state — events, the model
conversation, files — leaves the machines as it is produced, so either can
die at any point and the session resumes where it was. Details:
[How it works](/background-agents/how-it-works).

## Relation to sandboxes

Sessions work inside ordinary [sandboxes](/sandboxes/overview), booted from
the workspace's latest build and managed for you. `oc session shell <id>`
opens the actual machine. For full control of the loop and the machine, use
the [Agent SDK](/agents/overview) instead. [Hosted
Agents](/agents-api/overview) remains the surface for chat-style agents on
managed cores and channels; Background Agents is for event-triggered,
unattended work.

## Properties

- Sessions start in seconds, because the workspace rebuilt itself on the
  last push.
- Models are addressed `provider/model` and switched with one command —
  effective on the next call within a provider, on the next turn across
  providers. Bring your own keys, including OpenRouter.
- Agents hold no tokens. The platform opens PRs and posts comments on their
  behalf, attributed to the requesting user.
- Sessions are two-way. Talk to an agent in a Slack thread, watch it work, and
  steer it; it asks when it needs a decision. See
  [Channels](/background-agents/channels).
